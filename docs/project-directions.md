# ACP DESKTOP 项目方向建议

本文档基于当前项目实现，整理 ACP DESKTOP 后续可以推进的几个产品与工程方向。目标不是一次性全部实现，而是为后续 roadmap、版本规划和 issue 拆分提供可评估的方向池。

## 当前定位

ACP DESKTOP 当前是一个面向 ACP Agent 的桌面客户端，核心价值在于把本地 Agent 启动、会话管理、聊天交互、权限确认、工具调用展示和任务计划追踪整合到一个桌面应用中。

从现有实现看，项目已经具备以下基础能力：

- 桌面壳：基于 Wails，支持无边框窗口、托盘和本地文件桥接。
- Agent 管理：支持多 Agent 配置、命令参数、环境变量和热更新。
- 会话工作流：支持创建、恢复、刷新、断开会话。
- ACP 通信：前端封装 JSON-RPC 桥接层，后端管理 Agent 子进程 stdin/stdout。
- 安全交互：支持权限确认和授权模式切换。
- 可视化反馈：支持工具调用、任务计划、思考区、通信监控等展示。

后续方向可以围绕“更好用的 Agent 工作台、更安全的本地执行、更清晰的过程可观测、更稳定的工程底座”展开。

## 方向一：多 Agent 工作台

### 目标

把当前的 Agent 配置和会话能力升级为一个更完整的本地 Agent 工作台，让用户可以稳定维护多个 Agent、多个工作区和多种启动策略。

### 适合解决的问题

- 用户需要同时维护 Claude Code、Codex、内部 Agent 或自定义 ACP Agent。
- 不同项目需要不同 cwd、环境变量、代理和启动参数。
- Agent 配置可编辑，但缺少模板化、校验和快速切换能力。

### 可落地功能

- Agent 配置模板：提供常用 Agent 的预设模板，并支持复制为自定义配置。
- 配置校验：启动前校验命令是否存在、参数是否为空、环境变量是否完整。
- 工作区 Profile：把 Agent、cwd、代理和授权模式组合为可复用 Profile。
- 最近使用组合：记录最近的 Agent + cwd + proxy 组合，提升新建会话效率。
- Agent 健康检查：启动前执行轻量探测，提前暴露命令不可用或依赖缺失。

### 技术抓手

- 扩展 `internal/config/config.go` 的配置结构和校验能力。
- 在 `WorkspaceSessionDialog.vue` 中增加 Profile / 最近使用组合入口。
- 在 `SettingsView.vue` 中补充模板、复制和配置检查操作。
- 在 `internal/agent/manager.go` 中增加启动前检查和更明确的错误分类。

### 优先级

高。它直接提升首次使用和日常启动体验，也能减少 Agent 启动失败带来的排查成本。

## 方向二：会话资产化与知识沉淀

### 目标

让会话不只是临时聊天窗口，而是可以被检索、复盘、复用和导出的工作资产。

### 适合解决的问题

- 长会话中工具调用、计划和结论容易丢失。
- 用户恢复会话后，需要快速理解历史上下文。
- 重要分析结果需要沉淀为文档、提交说明或任务记录。

### 可落地功能

- 会话摘要：为每个会话生成可编辑摘要，显示目标、关键结论、修改文件和未完成事项。
- 会话标签：支持按项目、任务类型、Agent、状态打标签。
- 历史搜索：按会话标题、消息内容、工具路径和计划项搜索。
- 导出能力：导出为 Markdown，包含消息、计划、工具调用和最终结论。
- 任务归档：把完成的 plan entries 保存为会话时间线。

### 技术抓手

- 扩展 `internal/store/store.go` 保存会话摘要、标签和索引字段。
- 在 `SessionList.vue` 增加搜索、过滤和标签展示。
- 在 `ChatMessages.vue` 或独立导出模块中实现 Markdown 导出格式。
- 在 `session.js` 中沉淀会话完成状态、计划历史和工具调用摘要。

### 优先级

中高。适合在基础会话稳定后推进，能够显著提升长期使用价值。

## 方向三：权限与本地安全治理

### 目标

把权限确认从单次弹窗升级为可审计、可配置、可回滚的本地安全策略体系。

### 适合解决的问题

- Agent 可能读写本地文件、执行命令或访问敏感路径。
- 用户需要在效率和安全之间做明确选择。
- 当前授权模式已存在，但还可以更细粒度、更可解释。

### 可落地功能

- 权限策略：按 Agent、工作区、工具类型、路径范围配置默认策略。
- 路径白名单/黑名单：对敏感目录、仓库外目录、系统目录做额外提示或阻止。
- 授权审计日志：记录每次工具授权、拒绝、自动放行和执行结果。
- 一次性授权与会话级授权：明确区分单次、本会话、长期记忆。
- 高风险操作提示：删除、覆盖、移动、执行 shell 等操作使用更强提示。

### 技术抓手

- 扩展 `frontend/src/lib/authorization.js` 的授权模型。
- 强化 `PermissionDialog.vue` 的风险分级和路径展示。
- 在 store 中记录授权事件，并提供审计视图。
- 在 Go 后端文件读写和 Agent 执行边界增加路径校验。

### 优先级

高。桌面端本地执行天然高风险，安全策略越早成型，后续功能扩张越稳。

## 方向四：可观测性与调试能力

### 目标

让用户和开发者更容易理解 ACP 通信、Agent 状态、工具执行和异常原因。

### 适合解决的问题

- Agent 启动失败、协议异常、超时和 stderr 日志需要更清晰地呈现。
- 当前已有 Traffic Monitor，但还可以成为完整的调试面板。
- 用户需要知道“现在卡在哪里、等待什么、失败原因是什么”。

### 可落地功能

- 连接状态时间线：展示 spawn、initialize、authenticate、session/new、prompt 等阶段。
- Traffic Monitor 增强：支持按 method、session、状态、耗时过滤。
- 错误诊断卡片：将常见错误映射为可操作建议。
- Agent stderr 面板：区分启动日志、警告、错误和协议输出。
- 性能指标：记录请求耗时、工具调用耗时、会话响应耗时。

### 技术抓手

- 扩展 `frontend/src/stores/traffic.js` 的事件模型。
- 在 `TrafficMonitor.vue` 中增加过滤、分组和耗时展示。
- 在 `internal/agent/manager.go` 中保留结构化进程事件。
- 在 `StartupProgress.vue` 中展示更细的启动阶段。

### 优先级

中高。对开发者和高级用户价值很高，也有助于项目自身排错。

## 方向五：桌面体验与系统集成

### 目标

强化桌面客户端区别于 Web UI 的体验优势，让常用操作更快、更贴近本地系统。

### 适合解决的问题

- 桌面应用需要更好的窗口、托盘、快捷键和本地文件体验。
- 用户可能长期后台运行，需要快速唤起、恢复和切换会话。
- 本地项目目录、文件变更和会话上下文可以更紧密连接。

### 可落地功能

- 全局快捷键：快速显示窗口、新建会话、聚焦输入框。
- 托盘增强：显示当前 Agent 状态、最近会话、快速断开。
- 工作区文件入口：在会话中快速打开 cwd、复制路径、查看最近工具访问文件。
- 窗口布局记忆：保存窗口大小、侧栏宽度、计划面板状态。
- 系统通知：长任务完成、权限等待、Agent 断开时通知用户。

### 技术抓手

- 扩展 `app/` 中托盘和窗口行为。
- 在 `internal/system/system.go` 增加系统能力封装。
- 在前端 store 中保存布局偏好。
- 通过 Wails runtime 能力补充窗口、剪贴板、通知和文件打开操作。

### 优先级

中。适合作为体验增强持续迭代，不必阻塞核心 Agent 工作流。

## 方向六：开放 API 与外部集成

### 目标

在桌面客户端内部能力稳定后，提供一组可控的本地 API，让外部工具、脚本、编辑器插件或自动化系统可以调用 ACP DESKTOP 的 Agent、会话和任务能力。

### 适合解决的问题

- 用户希望从 IDE、CLI、自动化脚本或内部平台触发 Agent 会话。
- 当前能力主要通过桌面 UI 使用，外部系统难以复用会话创建、Agent 调用和状态查询。
- 后续如果要开放生态，需要先定义稳定、受控、可审计的接口边界。

### 可落地功能

- 本地 HTTP API：提供只监听 `127.0.0.1` 的本地服务，暴露健康检查、Agent 列表、会话创建、消息发送、会话状态查询等接口。
- WebSocket/SSE 事件流：向外部订阅者推送会话更新、工具调用、权限等待、任务计划和完成事件。
- API Token：为本地 API 增加访问令牌，避免任意本机进程直接调用敏感能力。
- 权限桥接：外部 API 触发的高风险操作仍走现有权限确认和授权策略。
- OpenAPI 文档：生成接口文档，明确请求参数、响应结构、错误码和事件格式。
- API 开关：在设置页提供开启/关闭、端口配置、Token 重置和访问日志入口。

### 技术抓手

- 在 Go 后端增加独立 API server 模块，例如 `internal/api`，生命周期由 Wails App 管理。
- 将现有 Wails 绑定能力拆出可复用 service 层，避免 API server 直接耦合 UI 方法。
- 为会话、Agent、权限和事件流定义稳定 DTO，避免直接暴露前端 store 内部结构。
- 在 `internal/store` 中记录 API Token、端口、启用状态和访问日志。
- 在前端 `SettingsView.vue` 增加 API 开放设置面板。
- 在 `docs/` 中新增 API 设计文档和 OpenAPI 示例。

### 接口草案

首批 API 可以从低风险、可观测、易验证的接口开始：

```text
GET  /api/v1/health
GET  /api/v1/agents
GET  /api/v1/sessions
POST /api/v1/sessions
GET  /api/v1/sessions/{sessionId}
POST /api/v1/sessions/{sessionId}/messages
POST /api/v1/sessions/{sessionId}/cancel
GET  /api/v1/events
```

事件流建议先覆盖：

- `session.created`
- `session.updated`
- `message.created`
- `tool.started`
- `tool.completed`
- `permission.required`
- `plan.updated`
- `agent.disconnected`

### 安全边界

开放 API 必须默认保守：

- 默认关闭，需要用户主动开启。
- 默认只监听 `127.0.0.1`，不监听公网地址。
- 所有写操作要求 API Token。
- 涉及文件写入、命令执行、删除、移动等操作必须复用权限确认体系。
- API 访问日志需要可查看、可清空。
- 后续如果支持远程访问，应单独设计认证、TLS、来源限制和审计策略。

### 优先级

中高。它不是第一阶段的基础能力，但会决定 ACP DESKTOP 能否从桌面客户端扩展为本地 Agent 能力底座。建议在权限策略和会话模型稳定后启动。

## 方向七：工程质量与组件体系

### 目标

降低后续功能开发成本，让前端组件、状态管理、协议桥接和 Go 后端边界更清晰。

### 适合解决的问题

- 聊天、权限、工具调用、会话管理会持续增长，组件需要保持可维护。
- ACP 协议和本地桥接逻辑需要稳定测试。
- UI 已有 UED 组件基础，可以继续体系化。

### 可落地功能

- 前端组件分层：区分业务容器、展示组件、UED 基础组件和协议适配层。
- UED 组件规范：补充 Button、Dialog、Popover、Picker、Table 的使用约束。
- 协议桥接测试：为 `acp-bridge.js` 增加 JSON-RPC request/response/notification 单元测试。
- Store 行为测试：覆盖 session 创建、恢复、取消、权限处理和计划更新。
- Go 后端测试：覆盖配置加载、Agent 启动参数拼装、store 读写和路径处理。

### 技术抓手

- 将复杂组件继续拆分，例如 Chat 消息项、工具调用项、权限详情块。
- 增加前端测试框架和基础测试命令。
- 为 Go 模块补充 `go test ./...` 的稳定测试集。
- 在 `docs/` 中维护 UED 组件使用规范和架构边界文档。

### 优先级

中高。它不一定直接增加功能，但会决定项目能否长期稳定迭代。

## 推荐路线

如果目标是尽快形成一个更可靠、更可用的桌面客户端，建议按以下顺序推进：

| 阶段 | 时间范围 | 主线 | 关键交付 |
| --- | --- | --- | --- |
| 第一阶段 | 1-2 周 | 稳定启动与安全边界 | Agent 配置校验、权限策略雏形、错误诊断优化 |
| 第二阶段 | 2-4 周 | 会话体验增强 | 会话摘要、历史搜索、导出 Markdown、计划历史 |
| 第三阶段 | 4-6 周 | 可观测性增强 | 通信过滤、阶段时间线、stderr 日志视图、耗时统计 |
| 第四阶段 | 6-8 周 | API 开放准备 | 本地 API server、Token、事件流、OpenAPI 草案 |
| 第五阶段 | 持续 | 工程体系化 | UED 规范、测试补齐、组件边界文档、Go 后端测试 |

## 近期可拆 issue

以下任务适合作为近期可执行 issue：

1. Agent 启动前配置校验：检查 command、args、cwd、env，并在 UI 中展示明确错误。
2. 授权策略模型梳理：定义 manual、auto-first、session-allow、deny 等策略边界。
3. 会话摘要字段落库：为 session metadata 增加 summary、tags、updatedAt、status。
4. Markdown 导出：导出当前会话消息、计划、工具调用和结论。
5. Traffic Monitor 过滤：支持按 method、状态、耗时区间过滤通信记录。
6. UED Picker 规范：统一 ModelPicker、ModePicker、授权选择和通用菜单选择器的交互。
7. 错误诊断卡片：对 Agent 启动失败、初始化超时、协议错误给出用户可读说明。
8. 本地 API server 草案：提供 health、agents、sessions、messages 等首批接口。
9. API Token 与开关：在设置页增加 API 启用状态、端口、Token 重置和访问日志入口。
10. 后端 store 测试：覆盖本地 JSON 读写、坏文件恢复和并发写入场景。

## 取舍建议

- 优先做安全和启动稳定性：这是桌面 Agent 客户端的信任基础。
- 不急于做复杂多窗口：先把单窗口会话和 Agent 管理做扎实。
- 会话沉淀要尽早设计数据结构：后补索引和导出会比早期预留更费成本。
- UI 组件要持续收敛：不要让每个业务组件重复实现下拉、弹窗、按钮和状态样式。
- 可观测性要服务排错：不要只做数据堆叠，优先把错误原因和下一步动作讲清楚。
- API 开放要晚于权限模型：先明确安全边界，再允许外部系统触发 Agent 能力。

## 推荐优先级矩阵

| 方向 | 用户价值 | 工程风险 | 推荐优先级 |
| --- | --- | --- | --- |
| 多 Agent 工作台 | 高 | 中 | P0 |
| 权限与本地安全治理 | 高 | 中 | P0 |
| 会话资产化与知识沉淀 | 高 | 中 | P1 |
| 可观测性与调试能力 | 中高 | 中 | P1 |
| 桌面体验与系统集成 | 中 | 中 | P2 |
| 开放 API 与外部集成 | 中高 | 中高 | P1 |
| 工程质量与组件体系 | 中高 | 低 | P1 |

## 结论

ACP DESKTOP 最值得强化的方向，是把“能连接 ACP Agent 的客户端”推进为“安全、稳定、可复盘的本地 Agent 工作台”。短期应聚焦 Agent 启动稳定性、权限策略和基础会话沉淀；中期补齐可观测性、导出、搜索和本地 API 开放能力；长期再推进更深的桌面系统集成和组件体系建设。
